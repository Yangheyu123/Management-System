## 一、项目背景

### 1.1 业务背景
传统住宅小区的物业管理长期依赖纸质表单与 Excel 电子表格进行日常运营，存在以下痛点：

- **信息孤岛**：小区、楼栋、房屋、业主等基础数据分散在多份表格中，查询与统计困难。
- **流程不规范**：业主报修、投诉处理依赖电话或口头沟通，缺乏状态跟踪与闭环管理。
- **收费混乱**：物业费、水电费、车位费等账单靠人工核算，易漏缴、错缴，对账成本高。
- **数据不透明**：管理者无法实时掌握收费率、工单处理时效、设备运行状态等关键指标。

### 1.2 建设目标
构建一套面向住宅小区物业的 B 端管理系统，将物业日常运营的 **「人、房、车、钱、事、设备」** 六大维度业务线上化、可统计、可追溯，提升物业管理效率与服务质量。系统采用前后端分离架构，后端 Spring Boot 2.7 + MyBatis 提供 RESTful 接口，前端 Vue3 + Vite + Element Plus 单页应用，数据库 MySQL 8。

### 1.3 适用范围
本系统适用于中型住宅小区（户数 500~5000 户）的物业公司内部使用，支撑物业经理、收费员、维修员、业主等多种角色的日常协作。

## 三、系统架构

系统采用 **前后端分离 + 单体应用** 架构。前端 SPA 通过 HTTP 调用后端 RESTful 接口，后端单体部署内部分层。后端端口 **8080**，前端开发端口 **5173**（Vite proxy 代理 `/api` 到 8080）。

```
┌──────────────────────────────────────────────────────────┐
│                       浏览器 (用户)                          │
└────────────────────────────┬─────────────────────────────┘
                             │ HTTP
┌────────────────────────────▼─────────────────────────────┐
│            前端  Vue3 + Vite + Element Plus                  │
│   路由 / 鉴权拦截 / axios 封装 / Pinia 状态 / ECharts 图表      │
└────────────────────────────┬─────────────────────────────┘
                             │ /api/**  (Vite 开发代理 / Nginx 生产转发)
┌────────────────────────────▼─────────────────────────────┐
│            后端  SpringBoot 2.7 (端口 8080)                  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Controller 控制层   参数校验 / 调用 Service          │  │
│  ├────────────────────────────────────────────────────┤  │
│  │  Service 业务层      业务逻辑 / 事务                  │  │
│  ├────────────────────────────────────────────────────┤  │
│  │  Mapper 数据访问层    MyBatis SQL                    │  │
│  └────────────────────────────────────────────────────┘  │
│  横切：JwtInterceptor 鉴权 / GlobalExceptionHandler 异常 /    │
│        JwtUtils / BaseEntity 审计字段                        │
└────────────────────────────┬─────────────────────────────┘
                             │ JDBC
┌────────────────────────────▼─────────────────────────────┐
│                     MySQL 8                                │
└──────────────────────────────────────────────────────────┘
```

**后端四层架构**（单向依赖）：
| 层 | 包 | 职责 | 规则 |
|----|----|------|------|
| 表现层 | `controller` | 接收请求、参数校验、调用 Service、组装 Result | 不写业务逻辑，不直接调 Mapper |
| 业务层 | `service` | 业务逻辑、事务控制（`@Transactional rollbackFor=Exception.class`） | — |
| 数据访问层 | `mapper` | 执行 SQL，返回 Entity | 不含业务判断 |
| 实体层 | `entity`/`dto` | Entity 对应表，DTO 入参 | — |

辅助包：`common`(Result/PageResult)、`config`(WebMvcConfig/MybatisConfig)、`interceptor`(JwtInterceptor)、`utils`(JwtUtils/PasswordUtils/UserContext)、`exception`(BusinessException/GlobalExceptionHandler)。启动类 `com.demo.Application`。

## 四、用户角色说明

系统采用 RBAC（基于角色的访问控制）模型，预置 5 类角色：

| 角色 | 编码 | 职责描述 | 典型用户 |
|------|------|---------|---------|
| **超级管理员** | `SUPER_ADMIN` | 系统全部权限，负责系统配置、用户与权限管理、跨小区数据查看 | 物业公司 IT/系统管理员 |
| **物业经理** | `PROPERTY_MANAGER` | 负责本小区全部业务（基础数据、工单派发、收费审核、数据统计） | 小区物业经理 |
| **收费员** | `TOLL_COLLECTOR` | 负责账单生成、费用收取、缴费登记、收费报表查询 | 物业收费处工作人员 |
| **维修员** | `REPAIRMAN` | 负责接收工单、上门维修、完工反馈、设备巡检登记 | 物业维修班组 |
| **业主** | `OWNER` | 查看本人房屋、账单、提交报修工单、缴费 | 小区业主 |

**权限控制层级**：
- **菜单级**：登录后根据权限码动态渲染侧边栏菜单
- **按钮级**：通过 `v-permission` 指令控制页面按钮显隐
- **接口级**：后端接口通过 `@RequirePermission` 注解 + JWT 拦截器二次校验
- **数据级**：非超管用户受 `community_id` 约束，仅能查看所属小区数据

## 七、业务流程图

### 7.1 报修工单流程

```
业主提交报修                物业经理派单              维修员处理
    │                          │                       │
    ▼                          ▼                       ▼
[待派单] ──────────────▶ [已派单] ──────────────▶ [处理中]
    │                          │                       │
    │       业主撤销            │                     完工
    └──────────────▶ [已撤销]   │                       │
                                   ▼                       ▼
                              [已完成] ◀──────────────────┘
                                   │
                              业主评价 / 关闭
                                   ▼
                              [已关闭]
```

工单状态：1待派单 → 2已派单 → 3处理中 → 4已完成 → 5已关闭 / 6已撤销。每次状态变更均写入 `work_order_log` 操作日志表。

### 7.2 收费缴费流程

```
收费员生成账单              业主/收费员缴费            账单状态更新
    │                          │                       │
    ▼                          ▼                       ▼
按房屋+收费项目         记录实缴金额、            未缴 ─┬▶ 已缴清
批量生成应收账单  ───▶ 缴费登记          ───▶      │
（状态：未缴）            （写 payment）         部分缴纳
                                                  │
                                              错误账单作废 → 已作废
```

账单状态：1未缴 / 2部分缴纳 / 3已缴清 / 4已作废。一笔账单可多次部分缴纳，每次缴费写入 `payment` 流水。

### 7.3 用户登录与鉴权流程

```
用户输入账号密码
      │
      ▼
POST /api/auth/login
      │
      ▼
校验账号是否存在 ──否──▶ 用户名或密码错误
      │是
      ▼
BCrypt 比对密码 ──否──▶ 用户名或密码错误
      │是
      ▼
校验账号是否启用 ──否──▶ 账号已禁用
      │是
      ▼
生成 JWT Token（含 userId/username/roles），有效期 24h
      │
      ▼
返回 Token + 用户信息 + 权限码 + 菜单树
      │
      ▼
前端存储 Token，后续请求携带 Authorization: Bearer 头
      │
      ▼
后端 JwtInterceptor 校验 Token + @RequirePermission 权限校验
```

## 八、约束与假设

1. **假设**：物业公司已具备稳定的网络环境与 MySQL 数据库服务器。
2. **约束**：本系统不涉及业主端移动 App，业主功能通过 PC 端 Web 界面使用。
3. **范围限定**：本期不实现业主自助线上支付（对接微信/支付宝支付网关），缴费由收费员后台登记。
4. **数据迁移**：上线前需将存量小区/楼栋/房屋/业主数据录入系统。
